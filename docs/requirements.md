# 个人财务管理工具需求文档

## 1. 背景

这是一个面向个人使用的财务管理工具。它不是传统意义上只记录流水的“记账本”，而是用于管理个人现金流、账户余额、周期账单、信用卡账单、多币种资产和支付方式的个人财务面板。

当前阶段先不考虑银行同步、外部金融 API、NAS 集成或复杂投资管理。核心目标是通过高度定制化的手动录入，降低记账阻力，让用户可以用很少的点击完成日常记录。

## 2. 产品目标

1. 快速记录日常收入、支出、转账和账户余额变化。
2. 支持日元和人民币分别管理，并可按需查看折算后的总资产。
3. 区分“实际消费”和“账户间转移”，避免重复统计。
4. 支持信用卡账单周期，区分消费发生时间和实际扣款时间。
5. 支持 PayPay、微信、支付宝等余额型账户，记录充值、消费和剩余余额。
6. 支持固定周期支出、变动周期支出、随机支出、退款、分期付款和现金。
7. 通过模板、默认值和快捷入口优化手动录入体验。

## 3. 核心原则

### 3.1 手动优先，但尽量轻量

不依赖银行自动同步。系统应通过预设模板、默认账户、默认分类、周期提醒和快捷按钮，让用户每次只输入必要信息，通常只需要选择项目并输入金额。

### 3.2 账户和消费分离

一笔钱从银行充值到 PayPay，不代表已经花掉。只有从 PayPay 实际购买商品或服务时，才算作消费。

### 3.3 支付工具和资金来源分离

用户结账时看到的是支付工具，例如 Apple
Pay、PayPay、微信或支付宝。但资金最终可能来自信用卡、银行账户、钱包余额或现金。系统需要能表达这两层关系。

### 3.4 统计口径清晰

同一笔交易可能影响不同统计：

- 消费统计：按实际消费发生日期计算。
- 现金流统计：按账户实际出入账日期计算。
- 信用卡统计：按信用卡账单周期和扣款日期计算。
- 资产统计：按账户余额变化计算。

## 4. 用户财务结构

### 4.1 收入

当前主要收入较单一：

- 工资收入：每月一次，金额可变。
- 奖金收入：低频，可选。
- 其他收入：退款、报销、利息、转账收入等。

工资收入不需要高频入口。系统应每月提醒用户填写一次实际金额。

### 4.2 支出

支出按录入方式分为以下几类：

| 类型             | 示例                              | 金额 | 周期       | 推荐录入方式         |
| ---------------- | --------------------------------- | ---- | ---------- | -------------------- |
| 固定金额固定周期 | 房租、网络、订阅                  | 固定 | 固定       | 自动生成，用户确认   |
| 变动金额固定周期 | 水费、电费、燃气费、手机费        | 变动 | 固定       | 到期提醒，用户填金额 |
| 高频随机支出     | 超市、便利店、外食、交通          | 变动 | 随机       | 快捷按钮，输入金额   |
| 低频大额支出     | 家电、旅行、医疗、商场购物        | 变动 | 随机       | 普通表单，支持备注   |
| 网购支出         | Amazon、淘宝、京东、其他电商      | 变动 | 随机       | 快捷入口，标记平台   |
| 资金转移         | PayPay 充值、信用卡还款、现金取款 | 变动 | 随机或周期 | 账户转移，不计入消费 |

### 4.3 账户

账户表示钱当前所在的位置。初始可支持：

- 日元现金
- 日本银行账户
- 日元信用卡 A
- 日元信用卡 B
- PayPay
- 人民币银行卡
- 微信余额
- 支付宝余额
- 人民币现金

每个账户至少包含：

- 账户名称
- 币种：JPY 或 CNY
- 账户类型：现金、银行、信用卡、电子钱包、第三方支付余额等
- 当前余额
- 是否计入净资产
- 默认支付方式或关联支付方式

### 4.4 支付方式

支付方式表示用户实际结账时使用的方式。它可能直接对应账户，也可能只是账户外层的支付工具。

示例：

| 支付方式  | 实际资金来源             |
| --------- | ------------------------ |
| 信用卡 A  | 信用卡 A                 |
| 信用卡 B  | 信用卡 B                 |
| Apple Pay | 背后的某张信用卡         |
| PayPay    | PayPay 余额              |
| 微信支付  | 微信余额或人民币银行卡   |
| 支付宝    | 支付宝余额或人民币银行卡 |
| 现金      | 日元现金或人民币现金     |

交易记录中应尽量同时保存：

- 支付方式
- 资金来源账户
- 币种

## 5. 交易类型

### 5.1 收入

收入表示资产增加，通常来自外部来源。

示例：

- 工资入账
- 奖金入账
- 报销到账
- 退款到账
- 利息收入

### 5.2 支出

支出表示实际消费，应该进入分类统计。

示例：

- 超市购物
- 房租
- 水电燃气
- 订阅费用
- 外食
- 医疗
- 交通

### 5.3 转账

转账表示账户之间的资金移动，不应计入消费统计。

示例：

- 银行账户充值 PayPay
- 银行账户还信用卡
- 银行账户取现金
- 微信余额提现到银行卡
- 支付宝余额转到银行卡
- 人民币银行卡之间转账

转账需要记录：

- 转出账户
- 转入账户
- 转出金额和币种
- 转入金额和币种
- 手续费，如有
- 汇率，如涉及换汇

### 5.4 调整

调整用于处理漏记、余额校准、系统外变化等。

示例：

- 月末发现现金少了 1000 日元
- PayPay 实际余额和系统余额不一致
- 人民币账户因汇率或手续费出现差额

调整应单独标记，避免混入正常消费分析。

## 6. 信用卡需求

信用卡是重点对象，因为它有消费日期、账单周期和实际扣款日期三层时间。

### 6.1 信用卡交易

信用卡消费应按消费发生日期计入支出分类，但不会立刻减少银行账户余额。

每笔信用卡消费需要记录：

- 消费日期
- 入账日期，可选
- 金额
- 币种
- 分类
- 使用的信用卡
- 所属账单周期
- 预计扣款日期

### 6.2 信用卡账单

系统应支持按信用卡维护账单信息：

- 账单周期开始日
- 账单周期结束日
- 扣款日
- 本期消费总额
- 本期已确认金额
- 是否已还款
- 实际还款账户

### 6.3 信用卡还款

信用卡还款应被视为转账，而不是支出。

原因：

- 消费已经在刷卡当天计入支出。
- 还款只是银行账户向信用卡账户结算债务。

系统需要避免“刷卡消费一次 + 还款再算一次”的重复统计。

## 7. 余额型账户需求

余额型账户包括 PayPay、微信余额、支付宝余额等。它们既可以接收充值，也可以直接消费。

### 7.1 充值

充值是转账，不是消费。

示例：

用户从银行账户给 PayPay 充值 2000 日元：

- 银行账户减少 2000 JPY
- PayPay 余额增加 2000 JPY
- 不计入消费统计

### 7.2 消费

使用余额型账户消费时，才计入支出。

示例：

用户用 PayPay 花费 1500 日元：

- PayPay 余额减少 1500 JPY
- 计入对应消费分类
- PayPay 账户仍剩余 500 JPY

### 7.3 余额校准

系统应允许用户定期输入 PayPay、微信、支付宝等实际余额。如果系统余额和实际余额不一致，可以生成调整记录。

## 8. 现金需求

现金需要作为独立账户管理。

### 8.1 现金取款

从银行账户取现金是转账，不是消费。

示例：

- 日本银行账户减少 10000 JPY
- 日元现金增加 10000 JPY
- 不计入消费统计

### 8.2 现金消费

实际使用现金购买商品或服务时，才计入支出。

### 8.3 现金余额校准

现金容易漏记。系统应支持定期校准现金余额，并将差额标记为调整。

## 9. 退款需求

退款需要追踪“应退款”和“已到账”两个状态。

### 9.1 退款记录

当用户发起退款时，可以从原支出创建退款追踪。

退款追踪信息包括：

- 原始交易
- 退款金额
- 预计到账账户
- 预计到账日期
- 当前状态：待退款、部分到账、已到账、取消
- 备注

### 9.2 退款到账

退款到账后，系统应记录一笔退款收入或负支出，并关联原始交易。

退款统计应支持两种视角：

- 原始消费月仍显示原消费，但标记后续退款。
- 实际到账月显示退款到账，影响当月现金流。

## 10. 分期付款需求

分期付款需要同时表达总消费和每期扣款。

### 10.1 分期交易

创建分期时应记录：

- 商品或服务名称
- 总金额
- 币种
- 分类
- 支付方式
- 分期期数
- 每期金额
- 首期日期
- 后续扣款周期
- 手续费或利息，如有

### 10.2 统计口径

分期付款可能有两种统计方式：

1. 按购买日期一次性计入总消费。
2. 按每期实际扣款日期计入消费。

第一版可以先提供用户级设置，默认建议按购买日期计入消费，按每期扣款影响现金流。

## 11. 多币种需求

系统需要同时管理日元和人民币。

### 11.1 币种

初始支持：

- JPY
- CNY

每个账户必须绑定一个主币种。

### 11.2 汇率

第一版不需要自动获取汇率。可以支持手动设置参考汇率，例如：

- 1 CNY = 21.5 JPY

资产总览可以显示：

- JPY 资产合计
- CNY 资产合计
- 折算后的总净资产

### 11.3 跨币种转账

如果涉及换汇，需要记录：

- 转出币种和金额
- 转入币种和金额
- 实际汇率
- 手续费

## 12. 分类和标签

### 12.1 一级分类

建议初始分类：

- 居住：房租、网络、水电燃气
- 饮食：超市、便利店、外食、咖啡
- 交通：电车、打车、机票
- 购物：日用品、衣服、电子产品
- 订阅：软件、娱乐、云服务
- 医疗：医院、药品
- 社交娱乐：聚餐、电影、游戏
- 旅行
- 手续费和税费
- 其他

### 12.2 标签

标签用于补充细节，避免分类过度细分。

示例：

- Amazon
- 淘宝
- 可报销
- 退款中
- 冲动消费
- 工作相关
- 朋友代付

## 13. 录入体验需求

### 13.1 快捷按钮

首页应提供常用快捷按钮，例如：

- 超市
- 便利店
- 外食
- 咖啡
- 交通
- 网购
- PayPay 消费
- 现金消费
- 其他

用户点击后，只需要输入金额，其他字段由模板自动填充。

### 13.2 模板

模板应支持：

- 默认交易类型
- 默认分类
- 默认账户
- 默认支付方式
- 默认币种
- 默认备注
- 默认金额，可选

### 13.3 周期项目

周期项目用于自动生成待确认交易。

类型包括：

- 固定金额固定周期：房租、网络、订阅
- 变动金额固定周期：水、电、燃气、手机费
- 固定收入周期：工资

周期项目生成后可以是“待确认”状态，由用户确认或修改金额。

### 13.4 临时快速记录

系统可以支持只记录金额的临时条目，之后再补全分类和账户。

适用场景：

- 当下不方便细填
- 一天结束后统一整理

## 14. 提醒需求

系统应支持以下提醒：

- 每月工资录入提醒
- 固定支出确认提醒
- 变动账单金额填写提醒
- 信用卡账单确认提醒
- 信用卡扣款日前提醒
- 退款预计到账提醒
- 分期付款下一期提醒
- 账户余额校准提醒

## 15. 统计和视图需求

### 15.1 首页概览

首页应展示：

- 本月收入
- 本月支出
- 本月结余
- 当前账户余额
- JPY 资产合计
- CNY 资产合计
- 折算净资产
- 待确认周期项
- 退款待到账
- 信用卡待还款

### 15.2 支出分析

支持按以下维度查看：

- 月份
- 分类
- 账户
- 支付方式
- 币种
- 标签

### 15.3 账户视图

每个账户应展示：

- 当前余额
- 最近交易
- 本月流入
- 本月流出
- 待处理项目

### 15.4 信用卡视图

每张信用卡应展示：

- 当前账单周期
- 本期消费
- 预计扣款日
- 已确认账单
- 未入账消费
- 历史账单

### 15.5 退款视图

展示：

- 待退款
- 部分到账
- 已到账
- 逾期未到账

## 16. 数据字段草案

### 16.1 交易记录

每笔交易建议包含：

- ID
- 日期
- 交易类型：收入、支出、转账、调整
- 金额
- 币种
- 分类
- 标签
- 支付方式
- 来源账户
- 目标账户，转账时使用
- 是否计入消费统计
- 是否计入现金流统计
- 关联周期项目
- 关联信用卡账单
- 关联退款
- 关联分期计划
- 备注
- 创建时间
- 更新时间

### 16.2 周期项目

- ID
- 名称

- ID
- 名称
- 账户类型
- 币种
- 当前余额
- 是否计入净资产
- 默认支付方式

### 16.4 信用卡

- ID
- 名称
- 币种
- 账单日
- 扣款日
- 还款账户

### 16.5 退款追踪

- 原始交易 ID

- 预计到账日期

### 16.6 分期计划

- ID
- 原始交易 ID

- 币种

- 首期日期
- 周期
- 已完成期数
- 状态
- 手续费或利息

## 17. 第一版范围

第一版建议优先实现：

1. 账户管理，支持 JPY 和 CNY。
2. 收入、支出、转账、调整四类交易。
3. 快捷支出模板。
4. 固定周期和变动周期项目。
5. 信用卡账单周期和还款转账。
6. PayPay、微信、支付宝等余额型账户的充值和消费。
7. 现金账户和余额校准。
8. 退款追踪。
9. 基础分期计划。
10. 首页财务概览。

## 18. 暂不考虑

以下能力暂时不进入第一版：

- 银行 API 同步
- 信用卡 API 同步
- 自动读取邮件账单
- 股票、基金、加密货币等投资资产管理
- 多用户或家庭协作
- 发票扫描 OCR
- 自动汇率更新
- 复杂预算系统
- 税务申报

## 19. 待确认问题

1. 信用卡消费统计是否默认按消费日期计入支出？
2. 分期付款是否默认按购买日计入总消费，还是按每期扣款计入消费？
3. Apple Pay 是否始终对应固定信用卡，还是需要每次选择背后卡片？
4. PayPay 的资金来源是否需要区分银行充值和信用卡充值？
5. 微信和支付宝是否主要作为余额账户管理，还是需要进一步区分绑定银行卡？
6. 是否需要为朋友代付、AA、借出借入设计专门状态？
7. 是否需要支持预算，还是先只做统计？
8. 是否需要月末结账流程，用于检查账户余额和补漏？
