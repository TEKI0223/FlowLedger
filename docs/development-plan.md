# FlowLedger 开发流程规划

最后更新：2026-05-20

## 1. 规划目标

这份文档用于把 `docs/requirements.md` 中的产品需求转化为可执行的开发流程。当前项目已经具备 Next.js
App Router、TypeScript、Drizzle ORM、SQLite、PWA
manifest 和首页静态雏形，下一步重点不是一次性做完所有功能，而是先建立可靠的财务领域核心，再逐步补齐录入、账单、提醒和统计体验。

第一阶段的判断标准是：用户可以在手机浏览器中稳定完成日常手动记账，账户余额、消费统计、转账、信用卡还款不会互相混淆。

## 2. 当前项目状态

已具备：

- Next.js 16、React 19、TypeScript 项目骨架。
- Drizzle + SQLite 依赖和配置。
- 基础数据库 schema：账户、支付方式、分类、交易、信用卡、周期项目、退款、分期。
- 首页静态 UI 原型，覆盖概览、快捷录入、最近记录、待处理事项、账户余额。
- Dockerfile、compose.yaml、PWA manifest 雏形。
- 完整需求文档：`docs/requirements.md`。

尚未具备：

- 数据库迁移产物、初始化种子数据和真实读写链路。
- 领域规则校验，例如支出、转账、调整、信用卡还款对余额和统计口径的影响。
- 页面路由、表单、Server Actions 或 API 层。
- 测试体系和可重复的本地验证流程。
- 开发任务拆分、验收标准和版本边界。

## 3. 默认产品假设

以下假设用于推进第一版开发，后续如果需求有变化再调整：

- 单用户，暂不做登录、多用户和权限系统。后续上云后通过 Vercel 项目本身的访问控制（或不暴露公网域名）保证仅自己访问。
- 数据库统一使用 SQLite 方言。开发期完全使用本地 better-sqlite3 + 本地文件，schema 稳定后（M6 完成）再切换到 Turso（libsql 的托管服务）。Drizzle
  schema 不需要改动，只切换 client。
- 部署目标：Vercel 免费档 + Turso 免费档。NAS / 自托管方案保留为后续选项，不进入第一版规划。
- 开发期不连接 Turso，避免反复迁移和清理云端测试数据。
- 金额在业务层统一使用最小货币单位存储和计算，例如 JPY 为日元整数、CNY 为分，避免浮点误差。现有 schema 中的
  `real` 金额字段建议在早期迁移前调整。
- 信用卡消费默认按消费发生日计入支出统计，还款记为转账。
- 分期付款默认按购买日一次性计入消费，每期扣款只影响现金流和账户余额。
- Apple Pay 第一版配置为默认映射到某张信用卡，交易时允许临时覆盖背后卡片（见 requirements §4.4）。
- 每个支付方式都有一个默认资金来源账户，交易录入时使用默认值，必要时覆盖。
- PayPay、微信、支付宝第一版优先作为余额账户管理，充值是转账，消费才计入支出。
- 调整交易的录入语义是"输入实际余额，系统算差额"，不让用户自己心算（见 requirements §5.4 / §7.3 /
  §8.3）。
- 标签由用户在交易录入时动态创建，不预置字典、不提供独立的标签管理页。
- 提醒第一版只通过首页"待处理事项"区呈现，不做 PWA push、邮件或系统通知。
- 预算、OCR、邮件账单读取、自动汇率、投资资产、多用户协作不进入第一版。

## 4. 开发原则

### 4.1 领域规则先于页面丰富度

财务应用最怕统计口径不清。开发顺序应优先保证交易入库、余额变化、统计口径和账单归属正确，再扩展更漂亮的分析页。

### 4.2 每个里程碑都要可运行

每完成一个里程碑，都应能通过 `npm run lint`、`npm run build`
和关键手动流程验证。不要长期停留在只有 schema 或只有 UI 的半成品状态。

### 4.3 录入体验围绕手机操作设计

第一版应默认面向手机浏览器使用。快捷按钮、金额输入、账户选择、日期默认值和确认动作都要减少点击。

### 4.4 转账和消费必须分离

充值 PayPay、信用卡还款、银行取现、钱包提现都属于转账，不进入消费统计。只有真实购买商品或服务才进入支出统计。

## 5. 建议技术边界

### 5.1 应用结构

建议逐步形成以下结构：

```text
src/app/                 App Router 页面、布局、表单入口
src/app/actions/         Server Actions，处理创建和更新
src/db/                  schema、连接、迁移、seed
src/domain/              财务领域类型、规则、计算函数
src/features/            按业务模块组织 UI 和服务代码
src/lib/                 通用工具，例如日期、金额格式化、id
docs/                    需求、开发计划、决策记录
```

### 5.2 数据访问策略

- 页面读取数据时优先走服务函数，避免在组件中散落 SQL。
- 写入操作优先使用 Server Actions，表单验证使用 Zod。
- 所有会改变余额的写入都通过领域函数计算影响，再持久化。
- 交易列表、统计卡片和账户详情都从同一套查询口径派生，避免重复实现。

### 5.3 金额和日期策略

- 金额字段建议存储为整数最小单位：`amountMinor`、`balanceMinor`。
- 日期字段建议区分：
  - `occurredOn`：消费或事件发生日。
  - `postedOn`：可选，信用卡入账日或实际到账日。
  - `cashflowOn`：实际现金流发生日，必要时由交易类型推导。
- 时间戳字段保留 ISO 字符串，业务日期使用 `YYYY-MM-DD`。

## 6. 里程碑路线图

### M0：项目基线和开发护栏

目标：让项目具备可重复开发、构建和迁移的基础。

任务：

- 确认 Next.js、Drizzle、SQLite 的本地启动流程。
- 补齐数据库连接文件、迁移目录和 seed 脚本。
- 调整金额字段为整数最小单位，避免后续迁移成本扩大。
- 建立基础分类、账户、支付方式、快捷模板 seed 数据。
- 明确 lint、build、db generate、db migrate 的日常命令。

验收标准：

- 新开发者执行 README 中步骤后可以启动项目。
- 本地数据库可以初始化并看到默认账户、分类和支付方式。
- `npm run lint` 和 `npm run build` 通过。

### M1：账户与基础交易闭环

目标：完成最小可用记账闭环。

任务：

- 实现账户列表、账户创建、账户编辑和余额展示。
- 实现收入、支出、转账、调整四类交易创建。
- 实现交易创建时的余额影响规则：
  - 收入：目标账户增加。
  - 支出：来源账户减少，并计入消费统计。
  - 转账：来源账户减少，目标账户增加，不计入消费统计。
  - 调整：单账户余额校准，不计入普通消费分析。
- 实现交易列表和交易详情。
- 实现基础分类选择、支付方式选择和备注。

验收标准：

- 可以手动创建一笔收入、一笔支出、一笔转账、一笔调整。
- 账户余额变化符合预期。
- 支出统计不包含转账和信用卡还款。
- 删除或编辑交易时，余额可以正确回滚并重新计算。

### M2：首页概览和快捷录入

目标：让日常高频记录足够快。

任务：

- 将首页静态数据替换为真实查询。
- 实现本月收入、本月支出、本月结余、JPY/CNY 资产、折算净资产。
- 实现快捷支出模板：
  - 超市
  - 便利店
  - 外食
  - 咖啡
  - 交通
  - 网购
  - PayPay 消费
  - 现金消费
- 快捷录入表单默认填充分类、账户、支付方式、币种和日期。
- 支持“临时记录”，只填金额，后续补全。

验收标准：

- 手机视口下，常用支出可以在少量操作内完成。
- 首页概览和最近记录会随真实交易变化。
- 快捷模板可配置，默认值不会写死在页面组件中。

### M3：周期项目和待确认流程

目标：覆盖工资、房租、订阅、水电燃气等周期场景。

任务：

- 实现周期项目 CRUD。
- 支持固定金额固定周期、变动金额固定周期、固定收入周期。
- 生成待确认交易草稿。
- 首页展示待确认周期项。
- 确认时允许修改金额、日期、账户、备注。

验收标准：

- 可以为房租生成固定金额支出。
- 可以为电费生成待填金额支出。
- 可以为工资生成待确认收入。
- 已确认项目进入正式交易，账户余额和统计同步更新。

### M4：信用卡账单周期

目标：把信用卡消费、账单确认和还款转账跑通。

任务：

- 完善信用卡账户和信用卡配置。
- 根据消费日期自动归属账单周期。
- 建立信用卡账单表或账单查询模型，展示本期消费、未入账消费、预计扣款日。
- 实现账单确认状态。
- 实现信用卡还款，写入银行账户到信用卡账户的转账。

验收标准：

- 信用卡消费按消费日期计入支出统计。
- 还款不重复计入支出。
- 信用卡视图能看到当前账单周期、本期消费、预计扣款日和历史账单。

### M5：余额型账户、现金和校准

目标：让 PayPay、微信、支付宝和现金使用自然。

任务：

- 明确钱包账户和现金账户的专用录入入口。
- 实现充值、提现、现金取款等转账模板。
- 实现余额校准流程，自动生成调整记录。
- 在账户详情页展示最近交易、本月流入、本月流出和待处理项目。

验收标准：

- PayPay 充值不会进入消费统计。
- PayPay 消费会减少 PayPay 余额并进入支出统计。
- 现金取款是转账，现金消费才是支出。
- 余额校准能留下可追踪的调整记录。

### M6：退款和分期

目标：覆盖低频但容易打乱统计的复杂交易。

任务：

- 实现从原交易创建退款追踪。
- 支持待退款、部分到账、已到账、取消状态。
- 退款到账时创建关联交易。
- 实现基础分期计划创建、期数、每期金额、手续费和完成状态。
- 在首页和退款视图展示待到账退款。

验收标准：

- 原始消费月仍能看到原消费和退款标记。
- 实际到账月能看到退款现金流。
- 分期计划能表达总消费和每期扣款。

### M6.5：迁移到 Vercel + Turso

目标：schema 基本稳定后再上云，避免开发期反复迁移和清理云端数据。

前置条件：M1-M6 的核心 schema（账户、交易、周期、信用卡账单、退款、分期）已经稳定，最近一段时间没有结构性变更。

任务：

- 安装 `@libsql/client`，把 `src/db/client.ts` 切到 `drizzle-orm/libsql`。
- 调整 `scripts/seed.mjs` 以兼容 libsql client。
- 本地开发 vs 生产通过环境变量切换（本地 file URL，生产 Turso URL + auth token）。
- 在 Turso 创建数据库，跑 Drizzle migration，导入 seed。
- Vercel 项目创建、环境变量配置、首次部署。
- 验证 M1-M6 全部流程在线上可用。

验收标准：

- 在手机浏览器访问 Vercel 域名能正常使用全部已实现功能。
- 同一份数据在手机和电脑上一致。
- 本地 `npm run dev` 仍可用，使用本地 SQLite 文件，与生产数据隔离。

### M7：统计分析和视图打磨

目标：把数据转化为可读的个人财务面板。

任务：

- 支出分析支持按月份、分类、账户、支付方式、币种、标签筛选。
- 账户详情视图完善。
- 退款视图完善。
- 信用卡视图完善。
- 增加手动汇率设置和折算净资产展示。
- 优化手机端表单、空状态、错误提示和加载状态。

验收标准：

- 用户可以回答“这个月钱花到哪里了”。
- 用户可以回答“现在各账户余额是多少”。
- 用户可以回答“哪些账单、退款、周期项还没处理”。

### M8：上线打磨与日常维护

目标：让 Vercel + Turso 部署稳定可长期自用。

任务：

- 明确升级迁移流程：本地写迁移、CI 或手动跑 `drizzle-kit` 推到 Turso。
- 增加 Turso 数据备份说明（`turso db dump` 定期导出到本地）。
- PWA 安装体验检查：添加到主屏幕、icon、splash。
- Vercel 域名访问控制确认（password protection 或自定义域名 + Cloudflare Access）。
- 撰写日常操作说明（自己用的维护手册）。

验收标准：

- 数据库迁移流程明确，能从开发环境推到 Turso。
- 备份脚本可用，能定期把数据 dump 到本地。
- 手机浏览器添加到主屏幕后可正常使用。
- 公网访问受控，不会被陌生人意外访问。

## 7. 开发节奏建议

建议按“领域核心 → 录入闭环 → 周期和信用卡 → 复杂交易 → 分析视图”的顺序推进。

每个功能模块采用同一套开发步骤：

1. 更新或确认数据模型。
2. 编写领域规则和 Zod 校验。
3. 编写数据访问函数。
4. 编写 Server Action 或 API。
5. 编写页面和表单。
6. 补充测试或手动验收清单。
7. 运行 lint、build 和关键流程验证。

## 8. 测试策略

### 8.1 单元测试优先覆盖领域规则

优先测试：

- 收入、支出、转账、调整对账户余额的影响。
- 信用卡消费和还款的统计口径。
- 周期项目生成日期。
- 退款到账的关联关系。
- 分期计划的金额拆分。
- 多币种折算。

### 8.2 集成测试覆盖关键流程

关键流程：

- 新建账户 → 创建支出 → 首页支出变化。
- 银行账户充值 PayPay → 消费统计不变，账户余额变化。
- 信用卡消费 → 账单周期归属 → 还款转账。
- 发起退款 → 退款到账 → 现金流变化。

### 8.3 手动验收保留真实使用场景

每个里程碑至少用以下设备宽度检查：

- 手机宽度：390px 左右。
- 平板或窄桌面：820px 左右。
- 桌面：1200px 左右。

## 10. 关键风险

### 10.1 金额精度风险

当前 schema 使用 `real`
保存金额，财务应用长期使用浮点数容易出现小数误差。建议在第一轮迁移前尽快改为整数最小单位。

### 10.2 余额回滚风险

如果交易支持编辑和删除，余额不能只在创建时简单增减。建议保留统一的交易影响计算函数，编辑时先撤销旧影响，再应用新影响；或者定期从交易流水重算账户余额。

### 10.3 信用卡统计重复风险

信用卡还款必须是转账，不是支出。所有支出分析默认使用消费发生日，现金流分析再使用实际扣款日。

### 10.4 功能范围膨胀风险

需求文档覆盖面很广，第一版不要加入银行同步、预算、投资资产、OCR、邮件读取和多用户。先把手动录入做到可信、快速、好用。

## 11. 待确认决策

以下问题需要在对应里程碑开始前确认：

- M2 前：快捷模板是否允许用户自定义排序和默认账户。
- 折算净资产已用 seed 中的 1 CNY = 21.5 JPY 打通最低形态。汇率编辑 UI 推到 M7 与统计打磨一起做。
- M3 前：周期生成是用户手动触发，还是进入首页时自动生成。
- M4 前：信用卡账单是否需要独立持久化账单表，还是先用查询模型计算。
- M5 前：PayPay 是否存在信用卡充值场景，是否需要额外统计（同 requirements §19）。
- M6 前：退款到账记为收入、负支出，还是提供用户级偏好。
- 后续：是否需要为朋友代付、AA、借出借入设计专门状态，是否需要月末结账向导（同 requirements §19）。

## 12. 推荐下一步

下一步建议从 M0 开始：

1. 调整金额模型。
2. 补齐数据库连接、迁移和 seed。
3. 定义交易影响规则。
4. 让账户和交易形成第一个真实闭环。

完成这个闭环后，FlowLedger 就会从静态原型变成真正可用的个人财务账本核心，后面的快捷录入、信用卡和统计都可以稳稳接上。
